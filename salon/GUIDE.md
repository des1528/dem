Этот документ нужен для подготовки к экзамену: ты должен будешь написать всё приложение **с нуля** самостоятельно. Здесь по шагам объясняется, что и почему мы делаем, и приведены мнемоники для запоминания.

## 0. Общая идея архитектуры

Мы пишем **минималистичное** приложение в **двух файлах**:

```
db.py    — три функции: conn(), fetch(sql, params), query(sql, params)
main.py  — всё остальное: окно, виджеты, обработчики
```

Намеренно нет классов, нет MVC, нет ORM. Это удобно потому, что:

- на экзамене ты пишешь по памяти — чем меньше абстракций, тем меньше шансов ошибиться,
- весь поток данных виден глазами: SQL → словарь → виджет,
- каждый блок кода — отдельная функция, и они мало связаны друг с другом.

**Главный приём, который нужно запомнить:**

```
db.fetch("SELECT ...", (params,))   →  список словарей
db.query("INSERT/UPDATE/DELETE ...", (params,))  →  выполняет и коммитит
```

Всегда передаём параметры через `%s` и кортеж — никакой склейки SQL через f-строки.

## 1. Третья нормальная форма (3НФ)

### 1.1 Три правила простыми словами

- **1НФ**: каждая ячейка хранит **одно** значение. Нельзя «телефоны через запятую», нельзя «Tag1, Tag2, Tag3».
- **2НФ**: соблюдается 1НФ, и каждый неключевой атрибут зависит **от всего** первичного ключа. Важна только тогда, когда ключ **составной**. Если первичный ключ — суррогатный `ID`, 2НФ выполняется автоматически.
- **3НФ**: соблюдается 2НФ, и **нет транзитивных зависимостей** — то есть никакой неключевой атрибут не определяется через другой неключевой атрибут.

### 1.2 Самый частый пример нарушения 3НФ

Допустим, в таблице клиентов было бы:

```
Client(ID, FirstName, LastName, GenderCode, GenderName)
```

`GenderName` зависит не от `Client.ID`, а от `GenderCode`. Это и есть транзитивная зависимость: `ID → GenderCode → GenderName`. Нарушение 3НФ.

**Лечение** — выносим справочник:

```
Gender(Code PK, Name)
Client(ID, ..., GenderCode FK → Gender.Code)
```

Теперь у клиента **только** код, а имя пола живёт в одной строке таблицы `Gender`. Если поменять «Мужской» на «М.», правка нужна в одном месте.

### 1.3 Как проверить любую таблицу за 30 секунд

1. Смотришь на каждый столбец: **«это одно значение или список?»** Если список — 1НФ нарушена.
2. Смотришь на каждый неключевой столбец: **«от чего он зависит — от ID или от другого столбца?»** Если от другого столбца — 3НФ нарушена, нужен справочник.
3. Связи многие-ко-многим (клиент ↔ теги) выносим в отдельную таблицу-связку (`TagOfClient`).

### 1.4 Наша схема

Схема из `my.sql` уже в 3НФ:

- `Gender` — справочник пола (выделен как раз ради 3НФ);
- `Service` — основная таблица услуг, все атрибуты зависят только от `ID`;
- `ServicePhoto` — отдельная таблица, потому что у одной услуги много фото (иначе была бы 1НФ-проблема);
- `Client`, `ClientService` — клиенты и их записи на услуги;
- `Tag` + `TagOfClient` — теги клиентов через связку.

На защите можешь сказать: «Схема приведена к 3НФ: все справочники (`Gender`, `Tag`, `Manufacturer`) вынесены в отдельные таблицы; повторяющиеся группы (фото, теги) — в дочерние таблицы со ссылкой на родителя; составные ключи использованы только в таблицах-связках».

### 1.5 Возможные вопросы экзаменатора

- **«Покажите нормализацию»** → объясняешь Gender и ServicePhoto как два примера.
- **«Что было бы, если бы фото хранились прямо в Service через запятую?»** → нарушение 1НФ, нельзя индексировать, нельзя удалить одно фото отдельно.
- **«Зачем `Code` в Gender — это же буква, а не число?»** → потому что это естественный короткий ключ, FK-ссылки получаются читаемыми (`М`, `Ж`).

## 2. Файл `db.py`

```python
import mysql.connector
CFG = dict(host="localhost", user="root", password="root", database="salon")

def conn():
    return mysql.connector.connect(**CFG)

def fetch(sql, p=()):
    c = conn(); cur = c.cursor(dictionary=True)
    cur.execute(sql, p); r = cur.fetchall()
    c.close(); return r

def query(sql, p=()):
    c = conn(); cur = c.cursor()
    cur.execute(sql, p); c.commit()
    rid = cur.lastrowid; c.close(); return rid
```

**Запомнить три строки:**

1. `cursor(dictionary=True)` — без этого `fetch` вернёт кортежи и тебе придётся обращаться по индексам.
2. `cur.execute(sql, p)` — параметры всегда передаются вторым аргументом, не вписывай их в SQL вручную.
3. `c.commit()` — без коммита `INSERT/UPDATE/DELETE` не сохранится.

## 4. Список услуг (`reload` + `draw`)

### 4.1 `reload()` — собираем выборку

```python
rows = db.fetch("SELECT * FROM Service")
total = len(rows)
# поиск
q = search_var.get().lower()
if q:
    rows = [r for r in rows if q in r["Title"].lower() or q in (r["Description"] or "").lower()]
# фильтр
f = filter_var.get()
if f != "Все":
    lo, hi = map(int, f.split("-"))
    rows = [r for r in rows if lo <= (r["Discount"] or 0)*100 < hi]
# сортировка вручную
rev = sort_var.get().startswith("По убыванию")
rows = sorted(rows, key=lambda r: float(r["Cost"]), reverse=rev)
draw(rows)
counter_label.config(text=f"{len(rows)} из {total}")
```

**Почему сортировка вручную, а не через `TreeView`?**
В ТЗ написано: «потому что в дальнейшем планируется усложнение этого функционала путем добавления дополнительных условий». То есть нам нужна **расширяемая** сортировка — `sorted(..., key=)`.

**Почему `(r["Discount"] or 0)*100`?**
Скидка в БД хранится дробью (0.30), а пользователь выбирает в процентах (30). Сравнение делаем в одних единицах.

**Граница включается слева, не справа: `lo <= x < hi`.**
Это прямо требование ТЗ. Скидка 30% попадает в `30-70`, не в `15-30`.

### 4.2 `draw(rows)` — рисуем карточки

Очищаем `services_frame`, заново строим `tk.Frame` на каждую строку. Фон зелёный — `#E8FFE8` — если скидка > 0.

**Перечёркнутая цена**:
```python
tk.Label(price, text=f"{cost:.0f} ₽", font=("Arial",10,"overstrike"))
```
`overstrike` — встроенный модификатор шрифта.

**Защита изображений от сборщика мусора:**
```python
lbl.img = photo
```
Иначе после выхода из функции `photo` будет собран GC и картинка пропадёт. Это **классическая ловушка** Tkinter — обязательно запомни.

### 4.3 Удаление с проверкой записей

```python
if db.fetch("SELECT 1 FROM ClientService WHERE ServiceID=%s LIMIT 1", (sid,)):
    return messagebox.showerror(...)
```

Сначала проверка — потом удаление дочерних `ServicePhoto`, потом самой услуги. Если есть хоть одна запись клиента — удаление **запрещено** (требование ТЗ).

## 5. Окно добавления/редактирования

Один и тот же `open_edit(sid)`:
- `sid=None` → добавление, поле ID скрыто;
- `sid=<int>` → редактирование, ID показывается только для чтения.

**Запрет на второе окно:**
```python
global edit_win
if edit_win and edit_win.winfo_exists():
    edit_win.lift(); return
```

### 5.1 Валидация (важно знать наизусть)

- Название не пустое.
- Уникальность: `SELECT 1 FROM Service WHERE Title=%s AND ID<>%s` — для нового `ID<>NULL` не сработает, поэтому собираем SQL и параметры по условию.
- Длительность `0 < dur <= 240` минут (ТЗ: не больше 4 часов и не отрицательная).
- Цена > 0, скидка в [0;1].

### 5.2 Изображение

`filedialog.askopenfilename` → копируем файл в локальную папку `images/` (требование ТЗ: «копируйте их в директорию с вашим проектом, чтобы пользователь мог удалить исходное изображение»). Сохраняем в БД **путь** к копии.

### 5.3 Дополнительные фото

Аналогично — `ServicePhoto`, удаление по кнопке-крестику. Для новой услуги фото можно добавлять только **после первого сохранения** (нужно знать `sid` для FK).

## 6. Запись клиента на услугу

Стандартная форма с тремя нюансами:

1. **`Combobox` клиентов** — словарь `FIO → ID`, чтобы по выбранной строке найти ID.
2. **`DateEntry` из tkcalendar** — если пакет не установлен, делаем обычный `Entry`. В коде это обёрнуто в `try/except`.
3. **Авторасчёт времени окончания** через `trace_add("write", upd_end)`. На каждое нажатие клавиши в поле времени пересчитывается метка «Окончание».

Валидация времени:
```python
re.match(r"^\d{1,2}:\d{2}$", time_v.get())
datetime.strptime(f"{date} {time}", "%Y-%m-%d %H:%M")
```

## 7. Окно «Ближайшие записи»

SQL — самый сложный в проекте:

```sql
SELECT cs.StartTime, s.Title, s.DurationInSeconds,
       c.LastName, c.FirstName, c.Patronymic, c.Email, c.Phone
FROM ClientService cs
JOIN Service s ON s.ID = cs.ServiceID
JOIN Client  c ON c.ID = cs.ClientID
WHERE DATE(cs.StartTime) IN (CURDATE(), CURDATE()+INTERVAL 1 DAY)
ORDER BY cs.StartTime
```

Расчёт «осталось N часов M минут»:
```python
mins = int((row["StartTime"] - datetime.now()).total_seconds() // 60)
h, m = divmod(mins, 60)
color = "red" if mins < 60 else "black"
```

**Автообновление каждые 30 секунд:**
```python
w.after(30000, refresh)
```
Метод `after` ставит **отложенный вызов** функции — это не блокирует UI, в отличие от `time.sleep`.

## 8. Режим админа

Самая простая часть:

```python
def toggle_admin():
    global IS_ADMIN
    if IS_ADMIN:
        IS_ADMIN = False
    else:
        code = simpledialog.askstring("Админ","Введите код:", show="*")
        if code != ADMIN_CODE: return messagebox.showerror("Ошибка","Неверный код")
        IS_ADMIN = True
    refresh_admin_ui()
    reload()
```

В `refresh_admin_ui()` показываем/прячем кнопки «+ Услуга» и «Ближайшие записи». В `draw()` проверяется `if IS_ADMIN:` чтобы добавить кнопки на карточку.

Это полностью соответствует ТЗ: «Список услуг должен быть виден всем в клиентской зоне, а функции добавления/удаления/редактирования и записи клиента — только администратору».

## 9. Реактивность поиска/фильтра/сортировки

Главный приём:

```python
search_var.trace_add("write", lambda *_: reload())
filter_var.trace_add("write", lambda *_: reload())
sort_var.trace_add("write",   lambda *_: reload())
```

`trace_add("write", ...)` вызывает функцию **каждый раз**, когда переменная меняется. Это и есть «фильтрация в реальном времени без кнопки найти».

## 11. Типичные ошибки и как их избежать

| Симптом | Причина | Решение |
|---|---|---|
| Картинки не показываются | GC съел `PhotoImage` | `lbl.img = photo` — держать ссылку |
| `1054: Unknown column 'Description'` | ошибка в названии колонки | имена столбцов в MySQL **регистрозависимы** на Linux |
| `IntegrityError: foreign key` | не заполнена справочная таблица | `INSERT INTO Gender ...` перед импортом клиентов |
| `mysql.connector.errors.InterfaceError: 2003` | MySQL не запущен / другой порт | проверь Workbench и `CFG` в db.py |
| Скроллбар не работает | забыли `canvas.configure(scrollregion=...)` | биндим `<Configure>` на внутренний Frame |
| После сохранения список не обновился | забыли `reload()` в конце | всегда `reload(); win.destroy()` |
| Второе окно редактирования открывается | не проверяешь `edit_win` | `if edit_win and edit_win.winfo_exists(): edit_win.lift(); return` |
| SQL-инъекция через поиск | склейка строк | всегда `%s` + кортеж параметров |
