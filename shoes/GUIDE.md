# GUIDE — Магазин обуви (ДЭ-2026, Модуль 1 + 2)

Этот гайд — выжимка приёмов, нужных на экзамене. Архитектура такая же, как в `salon/` — поэтому повторяющиеся вещи объяснены кратко, а уникальные для М1 (авторизация, роли, стили М2) — подробно.

## 2. БД и 3НФ

Главный приём — **«один атрибут на справочник»**. У товара указан не `ManufacturerName`, а `ManufacturerID → Manufacturer.ID`. Это и есть устранение транзитивной зависимости.

Связь **много-ко-многим** (один заказ ↔ много товаров) реализована через таблицу-связку `OrderItem(OrderID, ProductID, Qty)`. Без неё пришлось бы хранить «список артикулов через запятую» — это нарушение 1НФ.

Таблица `Role` тоже выделена отдельно — это позволяет переименовать роль в одном месте и не править все строки `User`.

## 3. Авторизация и роли

### 3.1 Окно входа

Стандартный шаблон:
```python
def show_login():
    for w in root.winfo_children(): w.destroy()
    # ... поля Логин / Пароль / кнопки «Войти» и «Войти как гость»
```

При входе:
```python
r = db.fetch("""SELECT u.ID, u.FIO, r.Name AS Role FROM User u
                JOIN Role r ON r.ID=u.RoleID
                WHERE u.Login=%s AND u.Password=%s""", (login, password))
```

Если строка найдена — кладём её в глобальный `USER` и зовём `show_main()`. Если нет — `messagebox.showerror`. Кнопка «Войти как гость» просто ставит `USER=None`.

### 3.2 Роли в `show_main()`

В начале:
```python
role = USER["Role"] if USER else "Гость"
can_filter = role in ("Менеджер","Администратор")
can_crud   = role == "Администратор"
can_orders = role in ("Менеджер","Администратор")
```

Дальше всё построение UI обусловлено этими тремя флагами:
- `if can_filter:` — показать поля поиска/фильтра/сортировки и привязать `trace_add`.
- `if can_crud:` — показать кнопки «+ Товар», «Изм.», «Удал.» в карточках.
- `if can_orders:` — показать кнопку «Заказы».

Это очень компактный паттерн «role-based UI» — главное не разбрасывать проверки по всему коду, а собрать в трёх булевых переменных в начале.

### 3.3 Выход

Кнопка «Выход» в шапке вызывает `show_login()`. Глобальный `USER` сбрасывается при следующем входе.

## 4. Стили М2

### 4.1 Константы наверху файла

```python
FONT       = ("Times New Roman", 11)
BG_MAIN    = "white"
BG_ACCENT  = "#7FFF00"
BG_ACTION  = "#00FA9A"
BG_DISCOUNT= "#2E8B57"
```

Везде в виджетах указываю `font=FONT, bg=BG_MAIN`. Это рутина, но без неё не пройдёшь экспертизу.

### 4.2 Логотип / иконка

```python
try: root.iconbitmap("Icon.ico")
except: pass
```

`try/except` — потому что в каких-то системах ico не подгружается.

### 4.3 Фон карточки по правилам

```python
disc = r["Discount"] or 0
if disc > 15:           bg = BG_DISCOUNT
elif r["Stock"] == 0:   bg = "#DDDDDD"    # «нет в наличии»
else:                   bg = BG_MAIN
fg = "white" if bg == BG_DISCOUNT else "black"
```

Один блок — все три правила (М2 «скидка > 15» + общее «остаток 0 = серый»).

## 5. Поиск / сортировка / фильтрация (только М/А)

```python
q = search_v.get().lower()
if q:
    rows = [r for r in rows if any(q in str(r[k] or "").lower()
            for k in ("Article","Name","Description","Manufacturer","Category","Unit"))]
if manuf_v.get() != "Все производители":
    rows = [r for r in rows if r["Manufacturer"] == manuf_v.get()]
rows = sorted(rows, key=lambda r: float(r["Price"]),
              reverse=sort_v.get().startswith("По убыванию"))
```

**Хитрость:** в выпадающий список производителей первым ставим `"Все производители"` — это сброс фильтра.

**Совместность:** сортировка применяется в самом конце, поэтому сохраняется и после фильтрации, и после поиска (требование ТЗ).

## 6. Окно редактирования товара (`open_edit`)

### 6.1 Справочники-комбобоксы

Для категории / производителя / поставщика делаем словарь `name → id` и обычный `Combobox` (не `state="readonly"`, чтобы можно было ввести новое значение).

При сохранении — `get_or_create`:
```python
def get_or_create(table, name, mp):
    if not name: return None
    if name in mp: return mp[name]
    return db.query(f"INSERT INTO {table}(Name) VALUES (%s)", (name,))
```

Один маленький метод заменяет три копии «искать-или-вставить». Запомни — это типовой DAO-приём.

### 6.2 Сжатие изображения до 300×200

ТЗ требует ограничения. PIL делает это в одну строку:
```python
im = Image.open(p); im.thumbnail((300,200))
dst = os.path.join(IMG_DIR, os.path.basename(p))
im.save(dst)
```

При замене картинки **удаляем старую** из папки:
```python
if old and os.path.exists(old) and old != dst:
    try: os.remove(old)
    except: pass
```

### 6.3 ID

При добавлении ID не показывается — поле просто не отрисовывается. При редактировании показывается как `Entry(state="readonly")`.

ТЗ говорит «автоматически вычисляется +1». В MySQL это `AUTO_INCREMENT` — БД сама сделает.

### 6.4 Запрет двух окон

```python
global edit_win
if edit_win and edit_win.winfo_exists(): return edit_win.lift()
```

Идентично salon.

## 7. Заказы

### 7.1 TreeView вместо карточек

Заказов обычно много, удобнее таблица. Колонки: дата, доставка, пункт, клиент, код, статус. `tree.insert("", "end", iid=oid, values=(...))` — `iid` ставим равным ID заказа, чтобы по `tree.selection()[0]` сразу получить ID.

### 7.2 Удаление

Сначала `OrderItem`, потом сам заказ — FK не пустит, если идти наоборот.

```python
db.query("DELETE FROM OrderItem WHERE OrderID=%s",(oid,))
db.query("DELETE FROM `Order` WHERE ID=%s",(oid,))
```

Имя `Order` в MySQL — зарезервированное слово, поэтому в кавычках-обратных-кавычках.

## 8. Импорт исходных данных (`import_data.py`)

Особенности файлов из ресурсов:
- `Заказ_import.xlsx` — поле «Артикул заказа» имеет формат `"А112Т4, 2, F635R4, 2"` (пары артикул/количество через запятую). Парсится `parts = articles.split(", ")`, шаг 2.
- `Пункты выдачи_import.xlsx` — **первая строка тоже адрес**, а не заголовок. Не пропускаем её.
- `user_import.xlsx` — «Роль сотрудника» текстом → `get_or_create("Role", role_name)`.
